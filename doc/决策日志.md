# TMS 架构与范围决策日志

记录一下我做 TMS 时的一些想法：需求当时说得不算细，代码和文档里哪些地方是「先这么定」、哪些先不做、以后有空再补什么。有变动就在下面追加日期和说明即可。

---

## 1. 需求写得模糊时怎么理解

一开始需求里没有单独写「项目 / 工作空间」和完整权限，但按 SaaS 来想，这两块迟早要补。我的做法是：别等产品图画完再改表，模型里尽量留出 **project / tenant 这类维度**（哪怕先可空），别只绑死一个 `userId`，不然以后上团队、企业客户会很痛。

权限也是一样。首版可以简单（甚至先不鉴权），但别在代码深处写死「永远只有一个用户、写死 userId」这种假设。接口外面留好以后挂登录、鉴权的位置，比以后拆代码省事。

做设计方案时稍微多留一点余地，提前做好冗余，预留后续扩展空间：表结构能加字段就别老删；接口错误格式统一一下；业务尽量收在 Service 里别堆在 Controller。以后业务改方向，不至于整片重写。

---

## 2. 架构上几个主决定

当前还是 **Spring Boot 单体**，图的是开发快、部署简单，事务也好处理。以后要扩容，应用尽量做成 **无状态**，会话、热点数据往 **Redis** 这类地方放，多实例才好加机器。

数据库这边，线上更倾向 **托管 MySQL**（主从、备份有人管），自己搭就要自己扛备份和故障。Redis、Kafka 若上生产，也尽量用云托管或成熟集群，别单机硬顶。

三高简单说下我的取向：

- **可用**：多实例 + 健康检查，挂了的实例从负载里摘掉；依赖的中间件要有降级心理预期（例如 Redis 挂了怎么办）。
- **并发**：列表分页、索引别偷懒；该缓存的缓存；写多冲突的地方用版本号、锁或排队，别全靠数据库扛。
- **性能**：大列表必须分页；SQL 和索引定期看一眼。

伸缩性：应用横向加 Pod；数据库真到瓶颈再考虑读写分离、分库分表，**分片键**最好能和租户 / 项目对齐，后面好拆。

Kafka 当作可选项：要异步、审计、多消费者再接，不必第一天就绑死。

---

## 3. 这轮明确没做的

- **Todo 多人实时协作（WebSocket 推送）**：要做就做完整（订阅、换页、多实例、网关超时），半吊子实时反而难维护，所以先砍了。
- **登录注册、完整鉴权**：和安全、令牌、审计绑在一起，产品没定 OAuth / 自建 / SSO 之前，做起来会是很大的一块，先上个简陋登录容易留坑，这轮没上。
- **大而全的租户 / 项目后台**：产品形态还会变，先保证数据结构和 API 能长出来，比先做一大套后台实在。

上面这些以后该补还补，只是当前里程碑不交付。

---

## 4. 时间充裕的话会往哪补

如果按「以后项目上千、访问量上来」来想，会优先考虑：

- 登录鉴权、审计日志、错误码 和 Gateway 打通。
- WebSocket + Redis 做实时推送（支持sticky connection）。
- 错误码、日志、链路追踪弄整齐一点。
- 缓存、读写分离、分库分表按实际需要再上。
- 对于TODO Management System 来说，修改记录也是必须的，追溯历史及变更人是不可缺少的一环。这一块后续也需要补齐。

体量真大了，租户隔离、限流、分片这些要当成一套容量规划来想，别等数据库单点扛不住了再救火。
